# ADR-003. Claude Haiku + Sonnet 혼용

- **Status**: Accepted
- **Date**: 2026-04-26

## 결정

작업 복잡도에 따라 모델을 분리:

- **Haiku**: 단순 작업 (분류, 태깅, Router Agent 라우팅 판단)
- **Sonnet**: 복잡 추론 (초안 생성, 플랫폼별 변환, Agent 의사결정)

## 이유

1. **비용 최적화** — 단순 작업에 Sonnet을 쓰면 불필요한 비용 발생. Haiku는 Sonnet 대비 토큰당 비용이 훨씬 저렴.
2. **응답 속도** — Haiku가 응답이 빠름. Router처럼 매 요청에 호출되는 노드는 지연이 사용자 경험에 직접 영향.
3. **품질 vs 비용 trade-off의 명시적 관리** — 어느 작업에 어떤 모델을 쓰는지 의식적으로 결정하고, 측정 가능하게 만듦.

## 대안

- **모든 작업에 Sonnet 사용** 탈락 — 비용·지연 측면에서 비효율. Multi-Agent에서 호출 횟수가 많아 누적 부담 큼.
- **모든 작업에 Haiku 사용** 탈락 — 복잡 추론(초안 생성, 플랫폼별 변환)에서 품질 부족.

## 결과 / Trade-off

- (+) 비용·지연·품질의 균형
- (+) 면접 어필: "비용 최적화 의식 + 측정 기반 모델 선택"
- (−) 모델별 인스턴스 관리 코드 복잡도 증가 (`ClaudeClient`를 모델별로 따로 인스턴스화)
- (−) Haiku → Sonnet 폴백 로직이 필요해질 수 있음 (Haiku 결과 신뢰도 낮을 때)
- 앞으로 신경 쓸 점: Day 12 ADR 작성 시 Sonnet vs Haiku **실측 비교표** 추가 (정답률, 비용, 지연)
