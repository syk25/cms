# ADR-003. Claude Haiku 전면 채택 (Sonnet vs Haiku 비교)

- **Status**: Accepted (Updated 2026-04-28)
- **Date**: 2026-04-26

## 결정

**전 Agent에 Claude Haiku 4.5 단독 채택.**  
초기 설계는 복잡 작업(초안 생성·플랫폼 변환)에 Sonnet, 단순 작업(분류·라우팅)에 Haiku를 배분하는 혼용 전략이었다.  
Day 7~12 구현 및 LLM-as-a-judge 평가 결과 Haiku의 품질이 MVP 요건을 충족함을 확인했고, 비용·지연 절감을 위해 Haiku 전면 채택으로 결정을 갱신한다.

## Sonnet vs Haiku 비교표

### 가격 (2026-04 Anthropic 공식 기준)

| 항목 | Claude Haiku 4.5 | Claude Sonnet 4.6 | 비율 |
|------|-----------------|------------------|------|
| 입력 토큰 단가 | $0.80 / MTok | $3.00 / MTok | Haiku ~3.75× 저렴 |
| 출력 토큰 단가 | $4.00 / MTok | $15.00 / MTok | Haiku ~3.75× 저렴 |

### 실제 호출 비용 추정 (이 프로젝트 기준)

| 시나리오 | 입력 토큰 | 출력 토큰 | Haiku 비용 | Sonnet 비용 | 절감 |
|---------|-----------|-----------|------------|------------|------|
| 분류 Agent (글감 1건) | ~300 | ~60 | $0.00048 | $0.00180 | 73% |
| Router Agent (질문 1회) | ~200 | ~40 | $0.00032 | $0.00120 | 73% |
| Co-writing 초안 (1회) | ~600 | ~800 | $0.00800 | $0.03000 | 73% |
| Distribute 변환 (3 노드) | ~1,500 | ~2,400 | $0.01080 | $0.04050 | 73% |
| **전체 워크플로우 1회** | ~2,600 | ~3,300 | **$0.0152** | **$0.0573** | **73%** |

> 토큰 수는 실제 API 응답 기준 추정값. 글감·원본 길이에 따라 ±30% 변동.

### 응답 지연 (실측, 로컬 네트워크 기준)

| Agent | Haiku 평균 | Sonnet 평균 (추정) | 비고 |
|-------|-----------|------------------|------|
| 분류 Agent | ~1.2s | ~3.0s | 글감 1건 분류 |
| Router Agent | ~0.8s | ~2.5s | 매 요청 호출, 지연 민감 |
| Co-writing 초안 | ~3.5s | ~8.0s | 800토큰 출력 기준 |
| Distribute Fan-out (3 병렬) | ~4.0s | ~10.0s | 가장 느린 노드 기준 |

> Sonnet 지연은 공개 벤치마크 기준 추정. 실측 미수행.

### 품질 평가 (LLM-as-a-judge)

| 작업 | Haiku 평균 점수 | 기준 |
|------|----------------|------|
| Co-writing 초안 | 3.8 / 5 | 일관성·완성도·주제 적합성 |
| 플랫폼 변환 (인스타·브런치·스레드) | 4.0 / 5 | 플랫폼 규칙 준수·가독성 |

MVP 요건 합격 기준(3.5 / 5) 충족 확인.

## 이유

1. **비용** — 전체 워크플로우 1회 기준 $0.015, Sonnet 대비 73% 절감. Multi-Agent는 노드 수만큼 호출이 누적되므로 모델 선택이 비용에 직접 영향.
2. **지연** — Router Agent는 매 요청에 호출됨. Haiku ~0.8s vs Sonnet ~2.5s 차이가 사용자 체감 응답성에 직결.
3. **품질** — LLM-as-a-judge 평가에서 MVP 합격 기준(3.5/5)을 상회. 초안 생성·플랫폼 변환 등 복잡 작업에서도 허용 범위 내 품질 확인.
4. **모델 관리 단순화** — 혼용 전략은 작업 복잡도에 따라 분기 로직이 필요하고 모델별 인스턴스 관리 비용이 생김. Haiku 단일화로 코드 복잡도 감소.

## 대안

- **복잡 작업 Sonnet, 단순 작업 Haiku 혼용** — 초기 설계 방향. 품질 상한을 높일 수 있으나, MVP 수준에서 Haiku로 충분했고 비용·지연·코드 복잡도 증가가 이득보다 큼.
- **모든 작업에 Sonnet** — 품질 최대화. 그러나 Haiku 대비 ~4× 비용 + ~3× 지연 증가. Multi-Agent에서 누적 부담이 커서 기각.
- **오픈소스 모델 (Ollama 등)** — 비용 0. 그러나 Fly.io 512MB VM에서 실행 불가, 품질 검증 미수행으로 기각.

## 결과 / Trade-off

- (+) 비용 73% 절감, 응답 지연 ~60% 단축
- (+) 단일 모델로 코드·의존성 단순화
- (+) 면접 어필: "측정 기반 모델 선택 + 비용·품질 trade-off 의식"
- (−) 복잡 추론(긴 원본 퇴고·맥락 깊은 대화)에서 Sonnet 대비 품질 열위 가능성. 프로덕션 전환 시 재평가 필요.
- (−) Haiku 품질 한계 도달 시 Sonnet 폴백 로직 없음 — 향후 개선 포인트.

## 향후 재평가 기준

- Co-writing 퇴고 라운드가 5회 이상 필요한 경우
- LLM-as-a-judge 점수가 3.5 미만으로 지속될 경우
- 사용자가 "너무 짧다", "맥락을 잃었다" 피드백이 반복될 경우
