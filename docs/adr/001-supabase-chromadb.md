# ADR-001. Supabase + ChromaDB 이중 저장 구조

- **Status**: Accepted
- **Date**: 2026-04-26

## 결정

원문은 Supabase(관계형 DB), 벡터는 ChromaDB(벡터 DB)에 분리 저장.
두 DB는 `source_id`로 연결.

## 이유

1. **저장소 적합성 분리** — 원문 수정·삭제·CRUD는 관계형 DB가 적합, 벡터 유사도 검색은 벡터 DB가 적합. 한 저장소에 두 작업을 강제하면 둘 다 비효율.
2. **검색 흐름의 자연스러움** — ChromaDB에서 유사 벡터 검색 → `source_id` 반환 → Supabase에서 원문 조회. 각 DB가 자기 강점만 발휘.
3. **확장성** — 벡터 모델 변경 시 ChromaDB만 재인덱싱하면 됨. 원문은 그대로.

## 대안

- **단일 벡터 DB(ChromaDB)에 원문 메타데이터까지 저장** 탈락 — 벡터 DB는 CRUD·트랜잭션 지원이 약함. 원문 수정 시 일관성 문제.
- **단일 관계형 DB에 벡터 컬럼 저장(pgvector)** 검토 — Supabase가 pgvector를 지원하므로 가능한 옵션. 14일 프로젝트 단계에서는 ChromaDB의 단순한 API와 Python 친화성이 학습·구현 속도 면에서 유리하다고 판단. 추후 통합 가능성은 열어둠.

## 결과 / Trade-off

- (+) 각 저장소가 강점에 집중, 확장성 확보
- (−) 두 DB 간 동기화 책임이 코드에 생김 (`source_id` 일관성 유지 필요)
- (−) 운영 복잡도 증가 (DB 두 개 관리)
- 앞으로 신경 쓸 점: Day 3~4 RAG 구현 시 동기화 실패 시나리오 처리 (원문 삭제 시 벡터도 같이 삭제되는지)
